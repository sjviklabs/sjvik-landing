01 Executive Overview

The SJVIK NOC is a production-grade operations center running on a 3-node Proxmox VE high-availability cluster. It provides centralized logging, SIEM, infrastructure-as-code, automated alerting, and a custom React-based operator dashboard for 13 LXC containers across 4 VLAN-segmented network zones.

Design Principles

  • Defense in depth: VLAN segmentation, SIEM with endpoint agents, least-privilege services, encrypted secrets at rest
  • Observable by default: Every container ships logs to a central aggregator, exports metrics to Prometheus, and is health-checked by an uptime monitor
  • Reproducible from code: Every container's state is captured in Ansible — rebuilding or provisioning a new LXC is a single command
  • Alertable: 13 Grafana rules cover disk, RAM, CPU, node health, backups, TLS certificates, and SIEM agent status

High Availability

5 critical containers are HA-registered in the Proxmox cluster with automatic restart policies. These cover DNS, reverse proxy, the operator dashboard, the automation agent system, and the monitoring stack. HA configuration uses max_restart 2 with no automatic relocation (manual failover only) to prevent split-brain scenarios during quorum recovery.

04 Network Segmentation

Traffic is segmented into 4 VLANs by function, enforced at the switch/bridge level. Cross-VLAN routing is controlled — the gaming VLAN is fully isolated with no access to management or service zones.

VLAN Purpose Access Policy Containers
Management DNS, proxy, monitoring, SIEM Full cluster access 7 containers
Services Applications, AI, automation Ingress from mgmt only 3 containers
Data Storage, secrets, sync, backup Restricted, no public ingress 5 containers
Gaming Game servers (isolated) No cross-VLAN access 1 container

Remote Access

  • Tailscale mesh VPN: Encrypted overlay network with dual subnet routers for failover. Primary and backup on separate physical nodes. Tested failover 2026-04-02.
  • Traefik reverse proxy: All internal services served via *.lan hostnames with locally-trusted TLS certificates (17 SANs).
  • AdGuard DNS: Local DNS rewrites for all service hostnames, plus network-wide ad/tracker filtering.

06 Infrastructure as Code

Every container's desired state is declared in Ansible. A single site.yml playbook converges all containers. New containers are provisioned with a one-command playbook that chains base roles automatically.

Repository Structure

infra-ansible/ ├── site.yml # Master playbook — converges everything ├── inventory/production/ │ ├── hosts.yml # 13 LXCs, 4 VLANs, 3 node groups │ ├── group_vars/ # Per-node and cluster-wide variables │ └── host_vars/ # Per-container log paths and overrides ├── roles/ │ ├── common/ # Base packages, SSH hardening, timezone │ ├── node_exporter/ # Prometheus metrics (idempotent install) │ ├── alloy/ # Log collection (Jinja2 → River/HCL) │ ├── 20+ service roles/ # One per container (adguard, traefik, ...) │ └── minecraft/ # Documented-only (isolated VLAN, manual mgmt) ├── playbooks/ │ ├── common.yml # Base roles for all containers │ ├── alloy.yml # Log agents (serial: 3 staggered deploy) │ ├── {node}.yml # Per-node service roles │ └── create-container.yml # One-command new LXC provisioning └── vault/ └── secrets.yml # AES-256 encrypted (Ansible Vault)

Key Practices

  • Audit-first role authoring: Every service role was written after a live SSH audit of the container's actual state — not from assumptions. This prevents "codifying drift" where the role captures stale or incorrect state.
  • Version-gated idempotency: Binary downloads use a version check (not file existence) to gate downloads. This eliminates spurious changed reports in check mode and ensures upgrades happen reliably.
  • Staggered deployments: The Alloy playbook uses serial: 3 to avoid overwhelming Loki's ingestion rate when 16 agents connect simultaneously.
  • Encrypted secrets: All credentials are encrypted with Ansible Vault. A pre-commit hook prevents accidental password file commits.
  • Conservative roles for critical services: Wazuh, PBS, and Vaultwarden roles manage service state only — they never attempt reinstallation, protecting complex single-package installs from Ansible-induced breakage.

08 Security Posture

SIEM — Wazuh

Full Wazuh deployment (indexer + manager + dashboard + filebeat) on a dedicated LXC with agents on production containers. Provides endpoint monitoring, file integrity monitoring, log analysis, vulnerability detection, and compliance reporting.

Network Security

  • VLAN segmentation: 4 zones isolate management, services, data, and gaming. The gaming VLAN is fully isolated — no cross-VLAN routing.
  • Encrypted mesh VPN: Tailscale with dual subnet routers on separate physical nodes for failover.
  • TLS everywhere: All internal services behind Traefik with locally-trusted certificates (17 SANs).
  • DNS filtering: AdGuard Home blocks ads, trackers, and malicious domains at the network level.

Access Control

  • SSH: Key-only authentication. Password authentication disabled on all containers. Root login restricted to key-based access.
  • Secrets management: Ansible Vault (AES-256) for all infrastructure credentials. Pre-commit hook prevents accidental exposure. Self-hosted Vaultwarden for operator password management.
  • Principle of least privilege: Services run as dedicated system users (nologin shell). Alloy log collectors use targeted group membership for read access rather than running as root.

09 Backup & Disaster Recovery

Backup Strategy

  • Proxmox Backup Server: Daily full backups of all 16 LXCs with deduplication
  • Grafana alerting: Automated alert fires if any container misses its backup window
  • Ansible as code backup: All container configs exist in the infra-ansible repo — a PBS restore + site.yml converge fully rebuilds any container

DR Validation

DR smoke test passed (April 2026): The automation agent container was restored from PBS to a different physical node. All 8 agents, scheduled jobs, and database were verified intact. Full recovery completed in under 10 minutes.

Redundancy

  • VPN failover: Dual subnet routers on separate physical nodes. Tested by killing the primary — all services remained reachable via backup.
  • HA restart: 5 critical containers auto-restart on failure (up to 2 attempts)
  • IaC rebuild: Any container can be recreated from Ansible playbooks after a PBS restore or from scratch

10 Monitoring & Alerting

Grafana Alert Rules (13)

Category Rules What It Catches
Disk 4 rules (warning + critical, legacy + new) Per-container disk usage at 70/80/85/90% thresholds
Memory 3 rules (warning + critical) Per-container RAM usage at 85/90/95% thresholds
CPU 1 rule Sustained CPU >90% across any container
Availability 2 rules Node unreachable, container unexpectedly stopped
Compliance 2 rules Backup missed, TLS certificate expiring (<30 days)
Security 1 rule Wazuh agent disconnected from SIEM manager

All alert rules are file-provisioned (GitOps-managed YAML in the Grafana provisioning directory), not manually created via UI. Alerts fire to an email contact point.

Origin story: The new disk/RAM alert rules were created after a cluster audit discovered 3 containers silently overflowing their disk allocations. The original alert only checked fstype="ext4", missing other filesystem types. The replacement uses a negation filter (fstype!~"tmpfs|overlay") to catch all real filesystems.

Additional Monitoring

  • Uptime Kuma: 11+ HTTP monitors covering all critical services at 60-second intervals
  • Grafana NOC Dashboard: Active container count, log volume by container, logs/minute rate, live log stream, log rate by node
  • Operator Dashboard /infra: Container cards, service health, backup status, alert summary — without needing Grafana access

11 Key Engineering Decisions

Every infrastructure decision has a documented rationale. These are the non-obvious choices that shaped the NOC.

Decision Rationale
Grafana Alloy over Promtail Promtail reached EOL Feb 2026. Alloy is the official successor with active development.
File-based log scraping, no journald Journal access fails silently in unprivileged LXC containers. File targets are reliable and explicit.
Loki ingestion rate raised to 32 MB/s Default 4 MB/s silently drops logs when 16 agents connect. Discovered during load testing.
TSDB v13 schema from day one Loki's schema is write-once. Starting with the latest schema avoids a painful future migration.
Ansible Vault over external secrets manager Zero-dependency encryption for a single-operator environment. Vault password file is gitignored with a pre-commit hook guard.
Conservative roles for complex services Wazuh, PBS, and Vaultwarden roles manage service state only — never reinstall. Prevents Ansible from breaking complex single-package deployments.
Staggered Alloy deployment (serial: 3) Deploying 16 log agents simultaneously could exceed Loki's burst ingestion limit. Staggering prevents silent log drops.
DNS server applied last in rollouts Applying Ansible to the DNS server (AdGuard) first caused a network-wide outage. Lesson learned: DNS is always last, with explicit health checks.
Express proxy for Loki queries Browser-to-Loki direct queries create CORS complexity and expose the log API. Express proxies all LogQL, keeping Loki internal-only.
Audit-first role authoring SSH-auditing each container before writing its Ansible role prevents "codifying drift" — where roles encode stale manual state as truth.
